## 1. 戦略の全体方針
*   **主目標**: 非指名オーガニック流入からの**診断開始数の最大化**
*   **副目標**: オーガニック月間10,000PV、回遊率改善、非指名キーワードのTop10獲得数増加
*   **アプローチ**: ビッグキーワードでの消耗戦を避け、ユーザーの深い悩み（ペイン）に答える**ロングテールキーワード**を徹底的に狙う。加えて、タイプ名を知らない検索者の**悩み起点検索**も積極的に取りにいく。
*   **フェーズ**: 100記事に到達した現在、主戦略を「記事量産」から**「既存記事の上位化・統合・回遊改善」**にシフトする。診断テストからバックエンド（マネタイズ等）への動線最適化は、月間1万アクセスが見えてきた段階（第2フェーズ以降）で検討する。
*   **累計記事数**: 100記事（執筆済み）。今後は110〜130記事を目標としつつ、リライトと設計改善の比率を高める。

### KPI構成

**主KPI**
- 非指名オーガニッククリック数
- オーガニック流入記事からの診断開始数
- 診断完了率

**副KPI**
- PV / UU
- 記事別平均エンゲージメント時間
- ページ/セッション（回遊率）
- Top10表示キーワード数
- クラスター別流入

## 2. なぜ「ロングテール（深い悩み）」を狙うのか？

1.  **競合回避と確実な上位表示**
    *   「INFP 特徴」「相性 16タイプ」などの1〜2語のビッグキーワードは、すでに大手メディアや古参ブログがドメインパワーで上位を占拠している。
    *   「INFP 仕事 続かない 原因」「ENTP 恋愛 飽きる 対策」といった3〜4語の掛け合わせキーワードは、検索ボリュームは少なくても競合が不在か弱いため、確実に1ページ目を取りにいける。
2.  **エンゲージメント（診断テスト実施率）の高さ**
    *   深い悩みを持つユーザーは、自分の悩みを解決してくれる記事を隅々まで熟読する。
    *   「あなたのその悩みは、〇〇という心理機能の偏りが原因です。まずは正確なタイプを診断してみましょう」という自然な文脈での誘導が極めて高いCVR（診断完了率）を生む。

## 3. 具体的な記事作成アクションプラン

### ① ターゲットの「リアルな悩み」の洗い出し

#### A. タイプ名起点（タイプ既知ユーザー向け）
各16タイプごとに、以下の4領域でキーワード（悩み）を発掘する。
*   **キャリア・仕事**: 適職、人間関係、モチベーション低下、転職の悩み
*   **恋愛・パートナーシップ**: コミュニケーションのすれ違い、理想と現実のギャップ
*   **メンタルヘルス**: ストレスの原因、バーンアウト（燃え尽き）、自己肯定感
*   **ライフスタイル**: お金の使い方、決断の先延ばし、習慣化

#### B. 悩み起点（タイプ未知ユーザー向け）
自分のタイプを知らない検索者を上流から取り込むため、以下のキーワードレイヤーを追加する。
*   **悩み×性格系**: 「仕事が続かない 性格」「人間関係 リセット癖 性格」「気を使いすぎて疲れる 性格」「恋愛で疲れやすい 性格」「飽きっぽい 性格 仕事」「決められない 性格」「一人が楽 性格」「断れない 性格」
*   **診断迷子系**: 「16タイプ 診断 しっくりこない」「MBTI 結果 毎回違う」「ソシオニクス MBTI どっち」「自分のタイプ わからない」
*   **比較・違い系**: 「16タイプとエニアグラムの違い」「内向型と人見知りの違い」「共感型と優柔不断の違い」

### ② 記事の基本構成テンプレート（悩み解決型）
すべての記事は以下の流れで構成し、ユーザーの満足度（検索意図の充足）を最大化する。
1.  **冒頭で答えを提示（40〜80字）**: 記事テーマに対する結論を最初に述べる。AI Overview時代においてSERPでの評価を得るためにも、冒頭に要点を置くことが必須。
2.  **共感と問題定義**: 「〇〇タイプ特有のこういう悩み、ありませんか？」
3.  **原因の解説（専門性）**: 「ソシオニクスや心理機能の観点から見ると、これは〇〇という機能が裏目に出ているからです」
4.  **解決策（アクション）**: 「具体的に明日からできる〇〇タイプ向けの対策」
5.  **ネクストアクション（診断誘導）**: 「納得できた方は、自分の傾向をより深く知るために無料の自己分析診断を受けてみてください」

#### 冒頭回答の例
> INFPが仕事を続けにくいのは、能力不足ではなく「価値観に合わない環境で消耗しやすい」からです。特に裁量の低い職場や感情を押し殺す対人環境では、強みが出る前に疲弊しやすくなります。

### ③ テンプレートの差別化
16タイプ展開で同一テンプレートをそのまま使うと「タイプ名だけ差し替えた量産記事」に見えやすい。各記事に最低でも以下のいずれか1つ以上を盛り込み、差別化を図る。
*   そのタイプ特有の失敗パターン
*   誤解されやすい行動例
*   逆効果になりやすい対策
*   似ている別タイプとの見分け方
*   短いケース例（あるある）
*   Aqsh Prisma独自の整理図

### ④ 独自性（E-E-A-T）の付与
Googleから「質の高いコンテンツ」として評価されるため、AIの一般的な回答だけでなく、以下の要素を必ず記事に盛り込む。（※これが `human-like-writing` スキルの腕の見せ所となる）
*   **経験（Experience）**: Aqsh Prisma独自の解釈や、具体的な「あるあるエピソード」の挿入
*   **専門性（Expertise）/ 権威性（Authoritativeness）**: MBTIやソシオニクスの理論的根拠（心理機能モデル）を明示し、「誰が書いているか（または監修しているか）」を明確にする
*   **信頼性（Trustworthiness）**: 「これは医療的アドバイスではなく、自己分析のフレームワークです」といった免責事項を設け、安全で誠実なサイト運営であることを明示する
*   まるで人間がカウンセリングしているかのような、温かみと説得力のあるトーン＆マナー

#### E-E-A-Tをサイト構造で見せる（追加固定ページ候補）
記事本文だけでは不十分。以下の固定ページを整備し、サイト全体の信頼性を担保する。
*   編集方針 / 記事制作フロー
*   監修・レビュー方針
*   参考文献・引用ポリシー / 訂正ポリシー
*   運営者情報
*   診断ロジック・理論の扱い方（「どの理論を主に使い、どこが一致し、どこが違うか」の明示）

### ⑤ YMYL / メンタルヘルス系記事の安全設計
「生きづらい」「疲れた」「燃え尽き」等のテーマはYMYL寄りのため、以下を必ず含める。
*   これは医療診断ではなく自己理解の補助である旨の明記
*   症状が強い・長引く場合は専門家相談を促す文の挿入
*   性格タイプで断定しない表現
*   「あなたが悪いのではない」系の安全な言い回し

#### 例文
> 強い抑うつ、不眠、食欲低下、希死念慮などがある場合は、性格タイプの問題として片づけず、医療機関や公的相談窓口への相談を優先してください。

## 4. 今後の進め方・KPI

### 記事制作の比率方針
100記事に到達した現在、新規追加だけでなく既存記事の上位化・統合・内部リンク最適化を重視する。以後は以下を目安とする。

| 区分 | 比率 | 月間目安本数 |
|---|---|---|
| **新規記事** | 30〜40% | 8〜12本 |
| **リライト** | 40〜50% | 8〜12本 |
| **統合・内部リンク・設計改善** | 20% | 3〜5本 |

### キーワード選定の基準
1.  **検索ボリュームの裏付け**
    *   **使用ツール**: Googleキーワードプランナー、ラッコキーワード、Ubersuggest（無料枠）
    *   **採用基準**: 月間検索ボリューム10〜500程度を目安とする。
2.  **ゼロボリューム特例（指名手配キーワード）**
    *   ツール上で「ボリューム０」と判定されても、**Googleサジェストや知恵袋に具体的な悩みとして存在する**のであれば、CVRが極めて高い「お宝キーワード」として積極的に採用する。
    *   主観が入りやすいため、以下のスコアで評価し**合計8点以上で採用**とする。

| 評価項目 | 3点（高） | 2点（中） | 1点（低） |
|---|---|---|---|
| 検索需要の痕跡（サジェスト/PAA/知恵袋/GSC） | 複数箇所で確認 | 1箇所で確認 | なし |
| 悩みの深さ | 切実な悩み | やや気になるレベル | 軽い興味 |
| 診断CTAとの近さ | 直接つながる | 間接的に関連 | 遠い |
| SERPの弱さ | 個人ブログ中心 | 混在 | 大手メディア占拠 |
| クラスターへの波及効果 | 複数記事を強化 | 1記事を強化 | 単独 |

3.  **1検索意図1URLの原則**
    *   同一SERPを争うキーワード群は1ページに統合し、意図が異なる場合のみ別ページを作成する。
    *   例（同一意図→統合）: 「INFP 仕事 続かない」「INFP 仕事 向いてない」「INFP 仕事 辛い」
    *   例（異なる意図→分離）: 「INFP 仕事 続かない」「INFP 恋愛 理想高い」
4.  **総合評価**
    *   キーワードは検索ボリュームだけでなく、検索意図の明確さ、SERPの弱さ、診断導線との近さ、クラスターへの波及効果を総合的に評価して採用する。

### 公開後のPDCA（リライトフロー）
公開後1〜2ヶ月でGoogle Search Consoleを確認し、以下の4パターンで判断する。

| ケース | 条件 | 対応 |
|---|---|---|
| **A: 最優先改善** | 表示回数多い × 順位8〜20位 | タイトル・導入文・冒頭回答の改善、内部リンク追加、関連セクション追記 |
| **B: ターゲット再定義** | 表示回数少ない × 順位20位以下 | キーワードミスマッチの検証、同意図ページとの統合検討 |
| **C: タイトル改善** | 順位高い × CTR低い | タイトル・ディスクリプション・日付・見出しの訴求力改善 |
| **D: コンテンツ改善** | CTR高い × 滞在短い | 記事の中身が検索意図に合致しているか検証、内容の充実化 |

### PV逆算目標（1記事あたりの目安）
月間1万PVを達成するための逆算。実際のPV分布はパレートの法則に従うため、「ヒット記事を意図的に生む構造」を意識する。

| 指標 | 目標値 | 備考 |
|---|---|---|
| 累計記事数 | 110〜130記事 | 現在100記事 |
| 月間PV目標 | 10,000PV | オーガニック検索のみ |
| 平均PV/記事 | 100PV/月 | 全記事の均等割り（理想値） |
| ヒット記事（柱の20%） | 20記事 × 400PV/月 = 8,000PV | ピラーページ等による「ミドルキーワード」の上位表示 |
| その他の記事（80%） | 80記事 × 25PV/月 = 2,000PV | クラスター記事による超ロングテールの底上げ |

**ヒット記事（月400PV）を生むための「面取り」条件:**
*   **ピラーページの役割**: ピラーページは**「ミドルキーワード（検索ボリューム1,000〜5,000）」**をターゲットとする。
*   **複数キーワードの面取り**: 1記事につき1つのキーワードに固執せず、LSIキーワード（共起語）やサジェストを網羅し、**1つの記事で3〜5個のロングテールキーワードの同時ランクイン**を狙う。ただし「1意図1URL」原則に従い、意図が異なるキーワードは別ページに分離する。
*   CTR目標は一律ではなく**順位帯別**に設定する（上位ほど高いCTRを期待）。

## 5. トピッククラスターモデル（内部リンク戦略）

個別のロングテール記事を「点」で量産するだけでは、サイト全体のSEO評価は上がりにくい。記事同士を**「トピッククラスター」**として構造的にリンクで束ねることで、Googleからの「このサイトはこのテーマに詳しい」という評価を獲得し、検索順位とユーザー回遊率の両方を底上げする。

### トピッククラスターの構造
*   **ピラーページ（柱記事）**: あるテーマを広くカバーする「まとめ記事」。各クラスター記事へのリンク一覧を掲載する。
*   **クラスターページ（個別記事）**: ピラーページのテーマに関連する、具体的な悩みに特化した記事。冒頭と結末でピラーページへリンクする。
*   **相互リンク**: ピラー↔クラスター間を双方向でリンクし、テーマの網羅性と一貫性を検索エンジンに伝える。

### 内部リンクの実装ルール
1.  各クラスター記事の**冒頭**（導入部）にピラーページへのリンクを自然な文脈で設置する
2.  各クラスター記事の**結末**（まとめ・ネクストアクション）にピラーページへのリンクを設置する
3.  ピラーページには、各クラスター記事への**リンク一覧セクション**を設置する
4.  同じクラスター内の記事同士も、関連性が高い場合は相互リンクする（例: #25 INFP仕事 ↔ #28 ISFJ断れない）
5.  **アンカーテキストの設計**: 意味のあるアンカーテキストを必須とする。完全一致だけでなく、**部分一致・文脈型アンカーも併用**し、自然で可読性の高い内部リンクを設計する。
    *   **完全一致**: 例「INFPが仕事を辞めたくなる理由」
    *   **部分一致**: 例「INFPが仕事で消耗しやすい理由」
    *   **文脈型**: 例「価値観と仕事内容のズレが苦しくなるパターン」

### オーファンページ対策
記事数が増えると内部リンクされていない記事（オーファンページ）が発生しやすい。以下のルールで防止する。
*   新規公開記事は**最低3本以上**の既存記事から内部リンクを設置する
*   各記事は**親ハブ1本 + 関連記事2本以上**のリンクを持つ
*   月1回、オーファン記事チェックを実施する

## 6. 診断導線の最適化

### ソシオニクスとエニアグラムの「役割分担」テンプレート
初めてサイトを訪れたユーザーが「どの診断を受ければいいか」で迷わないよう、記事内の誘導文脈で以下のステップを標準化する。

**標準テンプレート（記事内の誘導文）:**
> **Step 1**: まずは「16タイプ診断」で、自分の**思考のクセ（情報の処理パターン）**を特定しましょう。
> **Step 2**: さらに深い動機や行動パターンを知りたい方は「エニアグラム診断」で、自分を突き動かす**心のエンジン**のタイプを確認してみてください。

*   **使い分けの説明**: 16タイプ（ソシオニクス）は「情報の受け取り方・判断の仕方」、エニアグラムは「無意識の欲求・恐れ」をそれぞれ可視化するもの。両方を知ることで、自己理解が立体的になる。
*   **記事テーマによる使い分け**: キャリア・コミュニケーション系の記事では16タイプ診断を優先誘導、メンタルヘルス・モチベーション系の記事ではエニアグラム診断を優先誘導する。

### 検索意図（インテント）別の「CTA（診断誘導）」切り替え

記事末尾には共通のボタンコンポーネント `<DiagnosisCTA />` を設置する（テキストリンクは使用しない）。
ユーザーの検索意図によって刺さる言葉は異なるため、**本文のクロージング段落**のトーンを記事のインテントに応じて調整する。

*   **`[Know(知りたい)]` クエリの場合（基礎知識・仕組み・違い）**
    *   **心理**: 「情報を整理したい」「まだ自分の悩みが明確ではないが知見を得たい」
    *   **クロージング文の方向性**: 仕組みが見えてくると、自分の反応や他人とのズレにも一貫した説明がつく。まずは自分の思考パターンを整理してみよう、という流れ。
    *   **該当記事例**: ピラーページ全般、MBTIとの違いなど
*   **`[Do(行動したい・解決したい)]` クエリの場合（悩み・辞めたい・疲れた）**
    *   **心理**: 「原因を知り、今すぐこの現状を変えるためのアクションを起こしたい」
    *   **クロージング文の方向性**: この苦しさは「性格が弱いから」ではなく、思考の使い方と環境のミスマッチから起きている可能性がある。まずは自分の傾向を特定して、対策をズラさないことが大切、という流れ。
    *   **該当記事例**: 仕事辞めたい、恋愛疲れた、人間関係リセット癖など
    *   **本文中CTA（Do系のみ）**: 原因説明セクションの直後に、自然な文脈で診断への誘導を1回追加する。悩み系記事の読者は「原因がわかった瞬間」に最も行動しやすい。

> **実装メモ**: `<DiagnosisCTA />` コンポーネントは `src/features/articles/components/DiagnosisCTA.tsx` に実装されており、`src/app/articles/[slug]/page.tsx` の MDX コンポーネントマッピングに登録済み。ボタンテキストは「FREE ANALYSIS / 無料で診断する →」で固定。

### 診断「後」のリテンション（再訪）導線
PV/UU比を上げ、1人あたりの閲覧ページ数を増やすことで月間1万PV到達を加速する。

*   **診断結果ページ**: 「あなたのタイプにおすすめの記事」リンクを設置し、関連するクラスター記事への回遊を促す（将来的な実装課題）
*   **記事末尾**: 「同じタイプの他の記事を読む」セクションを設け、クラスター内の回遊を促進する
*   **期待効果**: 1人あたり平均2〜3ページの閲覧を実現し、同じユーザー数でもPVを倍増させる

### データ自動取得の仕組み（実装済み）

APIを使ったデータの自動取得スクリプトが構築済です。手動でGA4やGSCの管理画面を見に行く運用を避け、以下のアーキテクチャでデータを定期保存します。

#### 推奨アーキテクチャ
```
Google Analytics Data API (v1)
Google Search Console API (v1)
        ↓
  定期実行スクリプト（Python / Node.js）
  cron or GitHub Actions（週次/月次）
        ↓
  JSON / CSV として documents/analytics/ に保存
        ↓
  戦略ドキュメントのPDCAに活用
```

#### 実装ステップ
1.  **GCPプロジェクトでAPIを有効化**
    *   Google Analytics Data API
    *   Google Search Console API
2.  **サービスアカウントの作成とJSON鍵の取得**
    *   GA4のプロパティにサービスアカウントを「閲覧者」として追加
    *   GSCのプロパティにサービスアカウントを「閲覧者」として追加
3.  **データ取得スクリプトの作成**
    *   取得対象: 上記「KPIダッシュボード構成」の項目
    *   出力形式: JSON（プログラムで扱いやすい）+ CSVサマリ（人間が見やすい）
    *   保存先: `documents/analytics/YYYY-MM/` ディレクトリ
4.  **定期実行の設定**
    *   週次レポート: 毎週月曜にPV・UU・インデックス状況を取得
    *   月次レポート: 毎月1日にクエリ別・記事別の詳細データを取得
5.  **ダッシュボード（将来的）**
    *   取得データを元に、月間PV目標の進捗をMarkdownテーブルとして自動生成

#### 必要な環境変数
```bash
# .env に追加（絶対にGitにコミットしないこと）
GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account-key.json
GA4_PROPERTY_ID=properties/XXXXXXXXX
GSC_SITE_URL=https://your-site-url.com
```

> **⚠️ セキュリティ注意**: サービスアカウントのJSON鍵は `.env` または環境変数から読み込み、ソースコードやドキュメントに平文で記載しないこと。
