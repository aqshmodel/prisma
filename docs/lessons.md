## 1. 記事作成時の致命的なSEO・ガイドライン違反の防止

**【事象】**
時事トレンド記事5本の初稿作成時、以下の明確なガイドライン違反および「AI臭さ（人間らしさの欠如）」が指摘される状態となった。
- `description` の文字数不足（120〜160文字規定に対し、80〜90文字程度で短すぎた）
- `h1` `h2` `h3` の文字数制限（h1: 30-35字以内 / h2/h3: 13字以内）の超過。例: `## なぜ「好きな人」でも疲れるのか` (16字) など、カウントが甘かった。
- 内部リンクの配置ルール違反（「ピラーページへのリンクを冒頭と結末に必ず設置する」という指示を見落とし、本文中に適当に配置してしまった）。
- E-E-A-Tおよび「人間らしいライティング」の不足（文字数が少なく深みがない。詳細なペルソナ描写や、感情を揺さぶるストーリー体験が欠如し、単なる情報の羅列になっていた）。

**【今後の再発防止策と自分へのルール】**
1. **厳格な文字数カウント**: `description` とすべての見出しについて、目視ではなく厳格に文字数を数え、1文字でも超過・不足があれば再構成すること。
2. **ピラー誘導の徹底**: 全記事の導入（1段落目〜導入の終わり）および結論の段落に、ピラーページへのリンク（指定されたSEOターゲットKWをアンカーとする）を「息をするように」自然に確実に設置すること。
3. **文字量とストーリーの担保**: 短い1500文字程度の「あっさりした記事」は検索エンジンでも人間にも評価されない。最低でも2000〜3000文字のボリュームを持ち、ペルソナ（25歳事務職の〇〇など）の具体的なエピソードや心の動きをリアルに描くことで、人間のライター以上の血の通った文章を作成すること。
4. **自己レビュー**: 「作成を要求されたから作る」のではなく、「スタッフエンジニアがこれを承認するか？（ガイドラインと戦略に100%沿っているか）」を納品前に自問自答すること。
5. **AI臭さ（過剰な「」の排除）**: 強調やキーワードの囲みとして不必要に「」を多用するのはAIライティングの典型的な悪癖。実際の会話・引用以外は「」を削り、テキストとして自然に読める人間らしい文章を構築すること。

## 2. 新規記事ごとのupdatedAt必須化

**【事象】**
既存記事の修正時にのみ `updatedAt` フィールドを追記していたが、新規記事にも同様のフィールドを含めないと運用上の不整合が生じ、SEO的な更新日時の管理が漏れる要因となる。

**【今後の再発防止策と自分へのルール】**
- **新規作成であっても `updatedAt` を追加**: 記事を新規で執筆する段階（`date`フィールドの設定時）から、フロントマターに必ず `updatedAt: '2026-xx-xx'` を記述すること。
- その際、`date`（公開日時）と同一日付であっても省略せず、更新管理の初期値として必ず持たせること。

## 2026-03-06: 相性診断の回遊性向上（CTAとグローバルモーダル）
- **学び**: 診断結果のようなユーザーの関心（熱量）が最も高まるページでは、単なるナビゲーションリンクではなく、診断結果（ユーザーのType）に動的に連動した「具体的な相手（最高の相性・要注意な相性）」を提示するパーソナライズされたCTAが極めて有効。
- **UI/UXの改善**: 「最悪の相性」といったネガティブな表現を「Challenge Match（成長を促す相手）」にリフレーミングすることで、ユーザーの離脱を防ぎつつ前向きな興味を惹きつけることができる。
- **グローバルアクセスの設計**: コラムなどを読んでいる最中にもシームレスに使えるよう、ページ遷移を伴わないOverlay/Modal形式の検索UIを提供することで、没入感を削ぐことなくスムーズな導線を作れる。

## 2026-03-07: 記事作成時のガイドライン遵守と品質基準

**【事象】**
MBTI横断記事2本を新規作成した際、以下の重大な品質問題が発生し、3回のリライトを要した。
- 文字数が2,400字と、計画書で定めた4,000-5,000字を大幅に下回った（既存記事は13,000字超）。
- `title` のフォーマット（13字フック + 全体30-35字）、`h2/h3` の13字制限、冒頭回答40-80字、`slug` フィールドなど、執筆ガイドラインの基本仕様を複数違反した。
- カバー画像を誤ったディレクトリ（`/images/articles/`）に格納し、フロントマターの変数名も `image` と誤記（正しくは `coverImage`）。画像サイズも `640x360` で生成していた（正しくは `1200x675px`）。

**【今後の再発防止策と自分へのルール】**
1. **既存記事のfrontmatterをテンプレートとして必ずコピペ**: 新規記事を作るたびに、最新の既存記事（例: `mbti-result-changes-every-time.mdx`）のfrontmatterを丸ごとコピーし、値だけを書き換える。決して記憶だけで書かない。
2. **文字数は最低4,000字、目標6,000字以上**: 既存記事の実態（10,000-13,000字）を基準として意識する。2,000-3,000字の「あっさり記事」はSEO的にも人間的にも価値が低い。
3. **画像仕様は暗記**: `1200×675px / WebP / quality 75 / 100KB以下 / public/images/blog/{slug}.webp`。これは毎回確認するのではなく身体に染み込ませること。
4. **ガイドライン精読を初手で行う**: 記事を書き始める前に必ず `documents/プロジェクト戦略ver2/執筆ガイドライン.md` を読み直す。公開前チェックリスト（L116-L131）を1項目ずつ照合してからコミットする。
5. **推測で完了とせず、ツールで検証**: `check-guidelines.py` / `check-ai-patterns.py` / `wc -m` の3つを必ず実行し、基準を満たしていることを数値で確認してから報告する。

## 2026-03-08: カバー画像は必ず Nano Banana Pro で生成すること

**【事象】**
6本の記事のカバー画像を、Antigravity内蔵の `generate_image` ツールで妥協して作成してしまった。結果、クオリティが低い画像が生成された。ユーザールールに「画像は nano banana pro（`gemini-3-pro-image-preview` 等）を使用して生成し」と明記されているにもかかわらず、これに従わなかった。

**【今後の再発防止策と自分へのルール】**
1. **`generate_image` ツールでブログのカバー画像を作成することは厳禁**: 内蔵ツールは低品質であり、プロジェクトの基準を満たさない。
2. **必ず Nano Banana Pro（`gemini-3-pro-image-preview`）を使用する**: `scripts/archive/` 配下にある既存スクリプト（例: `generate-images-high-quality.py`）を参考に、Gemini API を直接呼び出すPythonスクリプトを作成・実行して画像を生成すること。
3. **画像仕様の再確認**: `1200×675px（横長）` / WebP / quality 75 / 100KB以下 / `public/images/blog/{slug}.webp`。アスペクト比は必ず横長（16:9相当）で最初から生成し、1:1で生成してから無理にリサイズしないこと。
4. **生成後にWebP変換**: `scripts/convert-to-webp.sh` を実行してWebP化と圧縮を行うこと。

## 2026-03-08: MBTI商標リスクの回避（16タイプ表記ルール）

**【事象】**
記事タイトルや本文中に「ESTp」「ISFj」「INTp」等の大文字4文字を多用していたが、これらはMBTI（マイヤーズ・ブリッグス財団）が商標として権利主張しており、法的リスクがある。

**【今後の再発防止策と自分へのルール】**
1. **大文字4文字（MBTI式）の使用禁止**: `ESTp`, `INFj` 等の大文字4文字はコンテンツ内で一切使用しない。
2. **ソシオニクス式表記を使用**: 末尾の1文字を小文字にする（例: `ESTp` → `ESTp`, `ISFj` → `ISFj`）。
3. **「MBTI」という語自体も避ける**: 代わりに「16タイプ性格診断」「性格タイプ」「ソシオニクス」等を使用する。
4. **新規記事を書く前にこのルールを確認する**: 無意識に大文字4文字を書いてしまいがちなので、毎回意識的にチェックすること。

## 2026-03-11: MDXファイルの frontmatter における slug 記載漏れと完全なYAMLブロックの徹底

**【事象】**
2026-03-10〜11にかけて作成した20記事分のMDXファイルにおいて、frontmatterに必須であるはずの `slug` フィールドが漏れていた。
原因は、事前の「記事構成案（実装計画）」を作成する段階で、`slug` や `category` を箇条書き（`- **slug**:` 等）で定義していたため、AIが実際のファイル生成時に標準的なMarkdownの知識（tagsで終わりがち）に引っ張られ、手動でYAMLブロックを組み立てる際に `slug` 行自体を出力し忘れたことにある。

**【今後の再発防止策と自分へのルール】**
1. **構成案の段階から完全なYAMLブロックを出力する**: 記事の構成案や実装計画をプランニングする時点で、メタデータを箇条書きで定義することを禁止する。必ず `---` で囲まれたコピペ可能な完全な形式（`title`, `description`, `date`, `updatedAt`, `coverImage`, `author`, `category`, `tags`, `slug` の全9項目）で出力する。
2. **執筆時のコピペ徹底**: 記事の実ファイルを生成する際は、構成案で作成したYAMLブロックを一文字も変更せず（内容だけ更新して）そのまま挿入すること。手動でフィールドを1行ずつ組み立てようとしないこと。

## 2026-03-12: MDXコンポーネントのグローバルインポートと目次の表示条件

**【事象】**
新規作成した10記事において、本文内に配置する `<DiagnosisCTA />` のために、各ファイルの先頭付近で `import { DiagnosisCTA } from '@/components/DiagnosisCTA'` を記述してコンポーネントエラー（不要なimportによる干渉等）を引き起こしかけた。
また、最新の10記事および過去の22記事において、目次（Table of Contents）が表示されないという不具合が発生した。

**【原因と仕様の理解不足】**
1. **MDXのグローバルコンポーネント化**: Next.jsの `MDXProvider` 等によって、`<DiagnosisCTA />` はグローバルに読み込まれる仕様となっていた。そのため、各記事のファイル内で個別の `import` 宣言は**一切不要**であり、単に本文中にタグを書くだけで機能する。
2. **目次（TOC）の表示条件**: 目次コンポーネント（`TableOfContents.tsx`）のレンダリング条件として、「`headings.length < 3` の場合は `return null`（非表示）」という仕様があった。つまり、**h2見出し（`##`）が最低3つ以上存在しないと、目次自体が生成されない**。これまでの記事はh2を2つにしてh3（`###`）で階層化していたため、この条件に引っかかり非表示となっていた。

**【今後の再発防止策と自分へのルール】**
1. **MDXファイル内での個別の `import` は禁止**: 既存の記事100本以上と同様に、コンポーネント（`<DiagnosisCTA />` など）を使用する際は `import` 文を書かず、タグだけを記述すること。システム全体のOS（グローバル設定）を事前に確認・理解してからコードを追加する癖をつける。
2. **h2見出しを必ず「3つ以上」配置する**: 目次を確実に表示させるため、記事の構成案（実装計画）を作成する段階で、大見出しである h2（`##`）を必ず3つ以上組み込む構造にすること。h3（`###`）だけで階層を深くしないこと。

## 2026-03-18: プロジェクトURLの誤認防止（works.aqsh.co.jp ではない）

**【事象】**
PR戦略のドキュメントやX投稿の設定などにおいて、Aqsh PrismaプロジェクトのURLを誤って `https://works.aqsh.co.jp` と認識・記載してしまっていた。正しくは `https://prisma.aqsh.co.jp` である。

**【今後の再発防止策と自分へのルール】**
1. **URLの絶対的な統一**: Aqsh Prismaプロジェクトのすべてのドキュメント、UTMリンク、キャラクター設定、フロントマターの参照先等における正しいベースURLは `https://prisma.aqsh.co.jp` である。今後一切 `works.aqsh.co.jp` と誤記しないこと。
2. **連携時の確認**: 外部サービス（X投稿アプリなど）と連携する際は、ベースURLが正しいサブドメイン（`prisma.`）になっているかを必ずチェックしてから設定を保存すること。
